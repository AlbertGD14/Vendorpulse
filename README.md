# VendorPulse

Multivendor SLI/SLO/SLA observability platform. Provides a unified web dashboard to monitor Service Level Indicators, evaluate Service Level Objectives and track Service Level Agreement compliance across multiple vendor services.

## Architecture

| Component | Description |
|---|---|
| **Prometheus** | Metric storage and rule evaluation engine |
| **Pyrra** | Generates Prometheus recording and alerting rules from SLO YAML definitions |
| **SLI Bridge** | Converts gauge metrics into Pyrra-compatible counters. Watches `/definitions/` and auto-detects new SLOs |
| **Backend API** | FastAPI service that queries Prometheus and exposes REST endpoints for the frontend |
| **Frontend** | React dashboard served via Nginx |
| **Vendor Services** | Simulated payments and catalog services exposing Prometheus metrics |

## Requirements

- [Docker](https://www.docker.com/)
- [Docker Compose](https://docs.docker.com/compose/)

├── services/           # Vendor services and SLI Bridge
├── prometheus.yml      # Prometheus scrape and rule configuration

## Usage

### Navigating the dashboard

Each service with at least one active SLO or SLA appears as a tab. Clicking a tab loads its SLOs (internal objectives) and SLAs (committed agreements), along with an availability history chart for the last 1h, 6h or 24h.

### Creating a SLO/SLA

Click **+ Create SLO/SLA** in the header to open the creation form. Fill in:

- **Service** — name of the service to monitor
- **Check** — name of the check (e.g. `latency`, `quality`, `throughput`)
- **Type** — SLO (internal, short window) or SLA (contractual, long window)
- **Target** — compliance percentage (e.g. `99` for SLO, `95` for SLA)
- **Window** — evaluation window (e.g. `1h`, `30d`)
- **Data source** — `Simulated` (Gaussian simulation) or `Real` (live Prometheus metrics)

For **Real** data source, additional fields appear:
- **Prometheus URL** — URL of the external Prometheus instance
- **Discover** — fetches available metrics from that Prometheus and populates the selector
- **Metric** — PromQL query identifying the gauge metric to monitor
- **Threshold / Condition** — the check passes if `value {condition} {threshold}` (e.g. `value <= 100`)

After submission the YAML is written to `/definitions/`. The SLI Bridge detects it within 30 seconds, restarts Pyrra and the new SLO appears in the dashboard automatically.

### Deleting a SLO/SLA

Hover over any SLO or SLA card and click the **✕** button. Confirm the deletion. The YAML definition and the generated Pyrra rules are removed and Prometheus reloads automatically. If the deleted entry was the last one for a service, the service tab disappears from the navigation.

### Adding a SLO manually

Create a YAML file in `definitions/` following the Pyrra specification:

```yaml
apiVersion: pyrra.dev/v1alpha1
kind: ServiceLevelObjective
metadata:
  name: my-service-latency
  labels:
    service: my-service
spec:
  description: "Latency check for my-service"
  target: "99"
  window: 1h
  indicator:
    ratio:
      errors:
        metric: sli_errors_total{service="my-service", check="latency"}
      total:
        metric: sli_checks_total{service="my-service", check="latency"}
```

For an SLA, add `sla` as a segment in the name (e.g. `my-service-sla-latency`) and use a longer window (`30d`).

For a **real metric** integration, add an `annotations` block under `metadata`:

```yaml
metadata:
  name: my-service-latency
  labels:
    service: my-service
  annotations:
    bridge.real: "true"
    bridge.prometheus_url: "http://host.docker.internal:9090"
    bridge.query: 'avg_p_latency{service_type="MyService"}'
    bridge.threshold: "100"
    bridge.condition: "<="
```

## Running the tests

The backend API has a test suite that runs without Docker or Prometheus. Install the dependencies and run pytest from the `vendorpulse/` directory:

```bash
pip install pytest httpx fastapi starlette
pytest tests/test_api.py -v
```

All 22 tests cover the main endpoints (`/api/services`, `/api/slo`, `/api/sla`, `/api/history`, `/api/slo/create`, `/api/slo/{name}` DELETE, `/api/prometheus/metrics`) using mocks for Prometheus queries and file system operations.

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `SLO_DIR` | `/definitions` | Path to the SLO/SLA YAML definitions directory |
| `PYRRA_RULES_DIR` | `/pyrra-rules` | Path to the Pyrra-generated recording rules directory |
| `CHECK_INTERVAL` | `15` | Seconds between each SLI Bridge simulation cycle |
| `WATCH_INTERVAL` | `30` | Seconds between each SLI Bridge directory scan for new definitions |
| `PYRRA_CONTAINER` | `pyrra_filesystem` | Name of the Pyrra container to restart when a new definition is detected |
